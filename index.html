Chapter 01

What Are Threads?

A thread is the smallest unit of CPU execution — a single sequence stream within a process, sometimes called a lightweight process.

🧵
Thread Definition
A thread is a single sequence stream within a process. Also called a lightweight process — threads allow multiple executions of streams within a single process while sharing resources.
⚡
Why Threads?
Threads improve application performance through parallelism. Multiple threads use CPU cores simultaneously, making programs faster, more responsive, and more efficient.
🔀
Concurrency vs Parallelism
Concurrency = dealing with multiple things at once (interleaving). Parallelism = doing multiple things truly simultaneously on multiple cores.
🏗️
Thread Structure
Each thread has its own stack and registers (PC, index register). But all threads in a process share code, data, and open files.

Thread Components Explained

Every thread has both shared and private components — understanding this distinction is fundamental.

📋
Code (Shared)
All threads execute within the same code segment. They can run same or different functions from the same binary. No duplication — lightweight!
🌐
Data (Shared)
Global variables and heap are shared among all threads. Easy communication — but requires synchronization (mutexes) to avoid race conditions.
📂
Files (Shared)
Open file descriptors are shared. All threads can read/write the same files. Closing a file in one thread affects all threads in that process.
🔧
Registers (Private)
Each thread has its own CPU registers: Program Counter (next instruction), Index Register, Stack Pointer, and general-purpose registers. Saved on context switch.
📦
Stack (Private)
Each thread's own call stack — stores function frames, local variables, and return addresses. Threads can call functions independently without interfering.
🆔
Thread ID (Private)
A unique identifier assigned by OS or thread library. Used to manage, target, join, or cancel specific threads during the process lifecycle.

Thread Classification: User vs Kernel

All threads are classified based on where they are created and managed.

👤 User Threads
Managed entirely in user space by a thread library. The OS kernel has no direct knowledge of them. Very fast to create and context-switch, but limited.
  • Managed by: POSIX Pthreads, Java threads, GNU Portable Threads
  • No kernel involvement → extremely fast context switching
  • One blocking system call blocks ALL user threads in process
  • Cannot achieve true parallelism in many-to-one model
  • Scheduler is entirely in user space (customizable)
  • Example: Solaris Green Threads (historical)
⚙️ Kernel Threads
Directly managed by the OS kernel. First-class citizens — the scheduler knows about each one and can distribute them across multiple CPU cores.
  • Created/managed by: the OS kernel directly
  • Heavier than user threads — kernel call overhead per operation
  • True parallelism on multicore systems ✓
  • One thread blocking does NOT block all others
  • Scheduler handles them natively (priority, CPU affinity)
  • Examples: Windows threads, Linux pthreads

4 Key Advantages of Multithreading

These four advantages make multithreading an essential tool in modern systems programming.

01

⚡ Responsiveness

Even if part of a process is blocked or doing heavy work, other threads continue. UI stays responsive while background threads load data — critical for interactive apps, games, and web servers.

02

🔗 Resource Sharing

Threads share memory and resources of the process by default. No complex IPC mechanisms needed — just shared variables. Far more efficient than process-to-process pipes or message queues.

03

💰 Economy

Creating and context-switching threads is ~30x cheaper than processes. Less memory overhead since threads share the address space. Thread pools reuse threads, eliminating repeated creation cost.

04

📈 Scalability

Multithreaded programs fully exploit multiprocessor architectures. Each thread can run on a different CPU core in true parallel. A 16-core machine can run 16 threads simultaneously.

Thread Lifecycle & States

Every thread transitions through well-defined states from creation to termination. Understanding these states is essential for debugging and synchronization.

BLOCKEDWaiting for I/O, lock, or resourceSystem call / lock contentionResource becomes available

Detailed State Descriptions

1. NEW (Created)

The thread has been created but has not yet started executing. The OS has allocated a TCB and initialized it. The thread is waiting for the scheduler to admit it to the ready queue.

2. RUNNABLE (Ready)

The thread is ready to execute and is waiting in the ready queue for CPU time.

3. RUNNING

The thread is currently executing on a CPU core.

4. BLOCKED (Waiting for I/O)

The thread is waiting for an I/O operation to complete.

5. WAITING (Timed/Indefinite)

The thread is waiting for a condition like wait() or sleep().

6. TERMINATED (Dead)

The thread has completed execution or was terminated.

💡
Context Switch

CPU switches between threads by saving/restoring state using TCB.

Thread Creation — In 3 Languages

Java — Two Approaches
// Approach 1: Extend Thread
class MyThread extends Thread {
    public void run() { System.out.println("Running: " + Thread.currentThread().getName()); }
}
new MyThread().start();  // ALWAYS use start(), not run()!

// Approach 2: Implement Runnable (preferred)
Thread t = new Thread(() -> System.out.println("Lambda thread!"));
t.start();

// Important thread methods:
t.join();        // Wait for t to finish
Thread.sleep(500); // Pause 500ms
t.interrupt();   // Request interruption
C — POSIX Pthreads
#include <pthread.h>

void* thread_func(void* arg) {
    int id = *(int*)arg;
    printf("Thread %d running\n", id);
    return NULL;
}

int main() {
    pthread_t threads[4];
    for (int i = 0; i < 4; i++)
        pthread_create(&threads[i], NULL, thread_func, &i);
    for (int i = 0; i < 4; i++)
        pthread_join(threads[i], NULL);  // Wait for all
}
Python — threading module
import threading

def worker(thread_id):
    for i in range(3):
        print(f"Thread {thread_id}: step {i}")

threads = [threading.Thread(target=worker, args=(i,)) for i in range(3)]
for t in threads: t.start()
for t in threads: t.join()  # Wait for all threads
Chapter 02

Synchronization & Concurrency Issues

The dark side of threads — race conditions, deadlocks, and the tools to tame them.

⚠️
Critical Issue: When multiple threads access shared data simultaneously and at least one modifies it, you get a race condition — unpredictable behavior, corrupted data, and bugs that only appear under specific timing conditions.
🏁
Race Condition
Outcome depends on thread execution order. E.g., two threads both read counter=5, both add 1 and write back 6 — final value is 6 instead of 7. The increment was lost.
🔒
Mutex / Lock
Mutual exclusion lock. Only one thread can hold it at a time. Others block until it's released. Prevents race conditions by serializing critical sections of code.
🚦
Semaphore
A counter-based synchronization tool. Binary semaphore (0/1) acts like a mutex. Counting semaphore limits concurrent access — e.g., only 3 threads can use a DB connection pool at once.
📡
Condition Variable / Monitor
Allows threads to wait for specific conditions. A thread calls wait() to sleep, another calls notify() to wake it up. Used in producer-consumer patterns — producer signals consumer when data is ready.
⚛️
Atomic Operations
Hardware-guaranteed indivisible operations. AtomicInteger in Java, std::atomic in C++. No lock needed — CPU handles the synchronization at the instruction level. Fastest option for simple counters.
📖
Read-Write Lock
Multiple readers allowed simultaneously, but only one writer at a time. Perfect for read-heavy scenarios like caches. Writers get exclusive access; readers share access among themselves.
Python — Race Condition vs Fixed
### ❌ WRONG — Race Condition
counter = 0
def increment():
    global counter
    for _ in range(100000):
        counter += 1  # NOT atomic! read-modify-write = 3 steps
# Expected: 200000 — Actual: ~153000 (lost updates!)

### ✅ CORRECT — With Lock
lock = threading.Lock()
def safe_increment():
    global counter
    for _ in range(100000):
        with lock:  # acquire + release automatically
            counter += 1
# Always exactly 200000 ✓

### ✅ Semaphore — limit 3 concurrent DB connections
db_sem = threading.Semaphore(3)
def query_db(thread_id):
    with db_sem:
        print(f"Thread {thread_id} using DB")
        time.sleep(1)

💀 Deadlock — Four Conditions

Deadlock occurs when two or more threads block forever waiting for each other. All four conditions must hold simultaneously.

Condition 1
Mutual Exclusion
A resource can only be held by one thread at a time. It cannot be shared simultaneously.
Condition 2
Hold and Wait
A thread holds one resource while waiting for another that is held by a different thread.
Condition 3
No Preemption
Resources cannot be forcibly taken from a thread. They must be voluntarily released.
Condition 4
Circular Wait
Thread A waits for Thread B, Thread B waits for Thread C, Thread C waits for Thread A → circle.
Python — Deadlock & Fix
### ❌ DEADLOCK
lock_A = threading.Lock(); lock_B = threading.Lock()

def thread1():
    lock_A.acquire(); time.sleep(0.1)
    lock_B.acquire()  # waits for B (held by thread2)

def thread2():
    lock_B.acquire(); time.sleep(0.1)
    lock_A.acquire()  # waits for A (held by thread1) → DEADLOCK!

### ✅ FIX — Always acquire locks in same order
def thread2_fixed():
    lock_A.acquire()  # A first — same order as thread1
    time.sleep(0.1)
    lock_B.acquire()  # B second — no circular wait ✓
    lock_B.release(); lock_A.release()

Producer-Consumer Pattern

Java — Thread-Safe Queue
class SharedQueue {
    private Queue<Integer> q = new LinkedList<>();
    private static final int MAX = 10;

    public synchronized void produce(int val) throws Exception {
        while (q.size() == MAX) wait();  // queue full, wait
        q.add(val);
        notifyAll();  // wake consumers
    }
    public synchronized int consume() throws Exception {
        while (q.isEmpty()) wait();  // nothing to consume
        int val = q.poll();
        notifyAll();  // wake producers
        return val;
    }
}
Chapter 03

Multithreading Models

How user-level threads map to kernel threads defines the model. Each has distinct tradeoffs in parallelism, overhead, and portability.

M:1
Many-to-One
Many user threads → 1 kernel thread

Many user threads managed by a single kernel thread. Fast thread management in user space — but one blocking call blocks all threads.

Advantages
  • Fast management
  • No kernel overhead
  • Portable
Disadvantages
  • One block = all block
  • No true parallelism
  • Rarely used now
Examples
Solaris Green ThreadsGNU Portable Threads
1:1
One-to-One
Each user thread → own kernel thread

Each user thread maps to its own kernel thread. True parallelism on multicore. One thread blocking doesn't affect others. Most widely used model today.

Advantages
  • True parallelism
  • Isolated blocking
  • Better concurrency
Disadvantages
  • Kernel overhead
  • Thread count limited
  • Heavy for many threads
Examples
WindowsLinux pthreadsSolaris 9+
M:M
Many-to-Many
Many user → Many kernel threads

Multiplexes many user threads onto a smaller set of kernel threads. OS creates sufficient kernel threads. Avoids limitations of both previous models.

Advantages
  • True parallelism
  • No thread limit
  • Efficient kernel use
Disadvantages
  • Complex to implement
  • Scheduler overhead
  • Less common today
Examples
Solaris <9Windows+ThreadFiber
2-L
Two-Level Model
M:M + direct binding option

Like M:M but also allows a user thread to be directly bound to a kernel thread. Critical threads get guaranteed CPU access via 1:1 binding, others share the pool.

Advantages
  • Most flexible
  • Priority binding
  • Best of both
Disadvantages
  • Most complex
  • OS-specific support
  • Legacy/rare
Examples
IRIXHP-UXTru64 UNIXSolaris 8−

🔒 Race Condition FixMedium
Fix the Race Condition
This code produces wrong results due to a race condition. Add a mutex to fix it — the answer should always be exactly 500,000.
counter = 0 # ❌ Race condition — NOT atomic! def increment(): global counter for _ in range(100000): counter += 1 # 3 steps: read, +1, write threads = [threading.Thread(target=increment) for _ in range(5)]
lock = threading.Lock() def safe_increment(): global counter for _ in range(100000): with lock: # acquire on enter, release on exit counter += 1 # Always 500000 ✓

Dining Philosophers
5 philosophers need 2 forks to eat. Implement a deadlock-free solution using semaphores and the asymmetric approach.
N = 5 forks = [threading.Semaphore(1) for _ in range(N)] def philosopher(i): left, right = forks[i], forks[(i+1)%N] # TODO: think → pick forks → eat → release # Hint: odd philosophers pick RIGHT first! ...
def philosopher(i): left, right = forks[i], forks[(i+1)%N] while True: time.sleep(0.1) # think # Asymmetric: even=left first, odd=right first f, s = (left,right) if i%2==0 else (right,left) f.acquire(); s.acquire() time.sleep(0.1) # eat s.release(); f.release() # ✓ no deadlock
🔍 Deadlock DetectionHard
Find & Fix the Deadlock
Identify the circular wait in this code and apply the lock-ordering fix to break the deadlock cycle.
lock_A = threading.Lock() lock_B = threading.Lock() def thread1(): lock_A.acquire(); time.sleep(0.05) lock_B.acquire() # waits for B… lock_B.release(); lock_A.release() def thread2(): lock_B.acquire(); time.sleep(0.05) lock_A.acquire() # waits for A → DEADLOCK!
# Fix: always acquire in same order (A before B) def thread2_fixed(): lock_A.acquire() # A first ← same as thread1 time.sleep(0.05) lock_B.acquire() # B second lock_B.release(); lock_A.release() # No circular wait → no deadlock ✓
🏊 Thread PoolMedium
Build a Thread Pool (M:M in practice)
Thread pools are the real-world implementation of Many-to-Many — N tasks run on M worker threads. This eliminates the cost of repeated thread creation.
from concurrent.futures import ThreadPoolExecutor def task(id): time.sleep(0.1) return id * id # 10 tasks, 4 worker threads (Many-to-Many!) with ThreadPoolExecutor(max_workers=4) as pool: futures = [pool.submit(task, i) for i in range(10)] results = [f.result() for f in futures] print(results) # [0,1,4,9,16,25,36,49,64,81]
💡
This is the Many-to-Many model in action — 10 tasks mapped to 4 threads. The pool scheduler manages assignment, just like an OS schedules threads to cores.
